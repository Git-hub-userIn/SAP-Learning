## 1. Project Setup

```bash
cds init <project-name>
cd <project-name>
cds add nodejs
# cds add hana
cds add hana --for production   # writes cds config block to package.json
```

> **Note:** `cds add hana` alone (on cds-dk >= 9.4.2) no longer writes the `cds` config block to `package.json`. You must run `cds add hana --for production` to get the config written.

Your `package.json` should end up with:

```json
{
  "dependencies": {
    "@cap-js/hana": "^2",
    "@sap/cds": "^9"
  },
  "devDependencies": {
    "@cap-js/sqlite": "^2",
    "@sap/cds-dk": "^9"
  },
  "cds": {
    "requires": {
      "[production]": {
        "db": "hana",
        "auth": "mocked"
      }
    }
  }
}
```

> **Important:** The `"auth": "mocked"` entry is required if you are not using XSUAA. Without it, CAP will auto-detect the CF environment and attempt JWT auth, which requires `@sap/xssec` and a bound XSUAA service. See [Error #4](#error-4-cannot-find-sapxssec) for details.

## 2. File Structure

```
db/
├── schema.cds                          ← CDS entities (CAP auto-generates hdbtable/hdbview)
├── data/
│   ├── study-Customers.csv             ← seed data
│   └── study-Orders.csv
└── src/                                ← native HANA artifacts (hand-authored)
    ├── .hdiconfig                      ← REQUIRED: registers HDI plugins for all file types
    └── get_order_summary.hdbprocedure  ← your procedure
srv/
├── catalog-service.cds                 ← service + function/action definitions
└── catalog-service.js                  ← handler that calls the procedure
package.json
mta.yaml
```

### What CAP auto-generates on `cds build --production`

| File | Location | Source |
|---|---|---|
| `.hdiconfig` | `gen/db/src/gen/` | Auto-generated by CAP |
| `.hdinamespace` | `gen/db/src/gen/` | Auto-generated by CAP |
| `*.hdbtable` | `gen/db/src/gen/` | From your `schema.cds` entities |
| `*.hdbview` | `gen/db/src/gen/` | From your service projections |
| `*.hdbtabledata` | `gen/db/src/gen/data/` | From your CSV files |
| Your `.hdbprocedure` | `gen/db/src/` | Copied as-is from `db/src/` |

> **Key insight:** CAP's auto-generated `.hdiconfig` only covers `gen/db/src/gen/`. Your hand-authored files in `db/src/` need their **own** `.hdiconfig` directly in `db/src/`.

## 5. Native HANA Artifacts

### 5.1 `.hdiconfig` — REQUIRED

**`db/src/.hdiconfig`**

This file must exist in `db/src/` and register the HDI plugin for every native artifact type you use. Use the full version below — it covers all common types and saves you from having to add entries later.

```json
{
  "file_suffixes": {
    "csv": { "plugin_name": "com.sap.hana.di.tabledata.source" },
    "hdbafllangprocedure": { "plugin_name": "com.sap.hana.di.afllangprocedure" },
    "hdbanalyticprivilege": { "plugin_name": "com.sap.hana.di.analyticprivilege" },
    "hdbcalculationview": { "plugin_name": "com.sap.hana.di.calculationview" },
    "hdbconstraint": { "plugin_name": "com.sap.hana.di.constraint" },
    "hdbfunction": { "plugin_name": "com.sap.hana.di.function" },
    "hdbindex": { "plugin_name": "com.sap.hana.di.index" },
    "hdbmigrationtable": { "plugin_name": "com.sap.hana.di.table.migration" },
    "hdbprocedure": { "plugin_name": "com.sap.hana.di.procedure" },
    "hdbrole": { "plugin_name": "com.sap.hana.di.role" },
    "hdbsequence": { "plugin_name": "com.sap.hana.di.sequence" },
    "hdbsynonym": { "plugin_name": "com.sap.hana.di.synonym" },
    "hdbsynonymconfig": { "plugin_name": "com.sap.hana.di.synonym.config" },
    "hdbtable": { "plugin_name": "com.sap.hana.di.table" },
    "hdbtabledata": { "plugin_name": "com.sap.hana.di.tabledata" },
    "hdbtabletype": { "plugin_name": "com.sap.hana.di.tabletype" },
    "hdbtrigger": { "plugin_name": "com.sap.hana.di.trigger" },
    "hdbview": { "plugin_name": "com.sap.hana.di.view" }
  }
}
```

> **Why is this needed?** CAP generates its own `.hdiconfig` at `gen/db/src/gen/` which only covers files it generates. Your native files in `db/src/` sit one level up — HDI cannot find the config and fails with `Could not find a ".hdiconfig" file in "src" or in any parent folder`.

### 5.2 The Procedure

**`db/src/get_order_summary.hdbprocedure`**

```sql
PROCEDURE "get_order_summary" (
    IN  iv_status  NVARCHAR(20),
    OUT ot_results TABLE (
        CUSTOMER_ID   NVARCHAR(36),
        CUSTOMER_NAME NVARCHAR(100),
        ORDER_COUNT   INTEGER,
        TOTAL_AMOUNT  DECIMAL(10,2)
    )
)
LANGUAGE SQLSCRIPT
SQL SECURITY INVOKER
READS SQL DATA
AS
BEGIN
    ot_results = SELECT
        c."ID"          AS "CUSTOMER_ID",
        c."NAME"        AS "CUSTOMER_NAME",
        COUNT(o."ID")   AS "ORDER_COUNT",
        SUM(o."AMOUNT") AS "TOTAL_AMOUNT"
    FROM "STUDY_ORDERS" AS o
    JOIN "STUDY_CUSTOMERS" AS c
      ON o."CUSTOMER_ID" = c."ID"
    WHERE o."STATUS" = :iv_status
    GROUP BY c."ID", c."NAME";
END
```

#### Table naming convention

CAP generates HANA table names by uppercasing the full CDS name and replacing dots and camelCase separators with underscores:

| CDS namespace + entity | HANA table name |
|---|---|
| `study.Orders` | `STUDY_ORDERS` |
| `study.Customers` | `STUDY_CUSTOMERS` |
| `my.orderShop.SalesOrders` | `MY_ORDERSHOP_SALESORDERS` |

> **You do NOT need a separate `.hdbtabletype` file** for `OUT TABLE` parameters. The table structure can be defined inline in the procedure signature as shown above.

## 6. Service Definition

**`srv/catalog-service.cds`**

```cds
using study as db from '../db/schema';

service CatalogService {
  entity Orders    as projection on db.Orders;
  entity Customers as projection on db.Customers;

  function getOrderSummary(status : String) returns array of {
    customer_id   : UUID;
    customer_name : String;
    order_count   : Integer;
    total_amount  : Decimal(10,2);
  };
}
```

> Use `function` (not `action`) for read-only operations. Use `action` for operations with side effects.

## 7. Service Handler

**`srv/catalog-service.js`**

```js
const cds = require('@sap/cds')

module.exports = class CatalogService extends cds.ApplicationService {
  async init() {
    const db = await cds.connect.to('db')
    const { Orders, Customers } = cds.entities('CatalogService')

    this.before(['CREATE', 'UPDATE'], Orders, async (req) => {
      console.log('Before CREATE/UPDATE Orders', req.data)
    })
    this.after('READ', Orders, async (orders) => {
      console.log('After READ Orders', orders)
    })
    this.before(['CREATE', 'UPDATE'], Customers, async (req) => {
      console.log('Before CREATE/UPDATE Customers', req.data)
    })
    this.after('READ', Customers, async (customers) => {
      console.log('After READ Customers', customers)
    })

    this.on('getOrderSummary', async (req) => {
      const { status } = req.data
      const result = await db.run(
        `CALL "get_order_summary"(iv_status => ?, ot_results => ?)`,
        [status]
      )
      // HANA returns OUT parameter keys in UPPERCASE
      return result.OT_RESULTS
    })

    return super.init()
  }
}
```

> **Note:** The handler stub from `cds add handler` generates `init()` without `async`. Change it to `async init()` and add `const db = await cds.connect.to('db')` at the top.

> **Local testing:** The `CALL` statement only works against a real HANA instance. It will fail with SQLite. For local testing, replace the procedure call with equivalent JS logic and swap back before deploying.

## 9. Errors & Solutions

### Error #1: `cds build` only shows `nodejs` task, no `hana` task

**Symptom:**
```
tasks: [
  { src: 'srv', for: 'nodejs', ... }
]
```

**Cause:** The `cds` config block is missing from `package.json`. `cds add hana` alone no longer writes it in cds-dk >= 9.4.2.

**Fix:**
```bash
cds add hana --for production
```

Then run `cds build --production`.

---

### Error #2: `Could not find a ".hdiconfig" file in "src" or in any parent folder`

**Symptom:**
```
Error: "src/get-order-summary.hdbprocedure": could not create a compile unit for the file
Error: Could not find a ".hdiconfig" file in "src" or in any parent folder
```

**Cause:** CAP's auto-generated `.hdiconfig` is in `src/gen/` — it does not cover files in `src/`. HDI requires an `.hdiconfig` in the same folder or a parent folder for every artifact.

**Fix:** Create `db/src/.hdiconfig` manually with the full plugin list (see [Section 5.1](#51-hdiconfig--required)).

---

### Error #3: `requires "db://STUDY_ORDERS" which is not provided by any file`

**Symptom:**
```
Error: "src/get-order-summary.hdbprocedure": the file requires "db://STUDY_ORDERS" which is not provided
```

**Cause:** Wrong table name in the procedure. CAP generates table names in **uppercase with underscores**, not with dots or camelCase.

**Fix:** Use uppercase underscore-separated names in your procedure SQL:
```sql
FROM "STUDY_ORDERS"   -- correct
FROM "study_Orders"   -- wrong
FROM "study.Orders"   -- wrong
```

---

### Error #4: `Cannot find '@sap/xssec'`

**Symptom:**
```
Error: Cannot find '@sap/xssec'. Make sure to install it with 'npm i @sap/xssec'
```

**Cause:** CAP auto-detects the CF environment (`VCAP_APPLICATION`) at runtime and switches to JWT auth as a default safety measure, even without an explicit XSUAA service binding or any auth config in `package.json`.

**Fix Option 1 — disable auth** (study/no-auth projects):

Add to `package.json`:
```json
"cds": {
  "requires": {
    "[production]": {
      "db": "hana",
      "auth": "mocked"
    }
  }
}
```

**Fix Option 2 — add XSUAA** (production-ready approach):

```bash
cds add xsuaa
npm add @sap/xssec
mbt build
cf deploy mta_archives/<project>_1.0.0.mtar
```

`cds add xsuaa` will:
- Add `@sap/xssec` to `package.json` dependencies
- Generate `xs-security.json`
- Add a XSUAA resource and binding to `mta.yaml`

This creates a `<project>-auth` XSUAA service instance on CF and satisfies the JWT auth requirement properly.

## 10. Local Testing Against HANA (Hybrid Profile)

Since `CALL` statements don't work with SQLite, you need to connect your local CAP app to the real HANA Cloud instance for testing procedures. CAP supports this via the **hybrid profile**.

### Step 1 — Log into CF
```bash
cf login -a <your-api-endpoint>
```

### Step 2 — Bind local project to the deployed HANA service
```bash
cds bind -2 procedures-db
```

This creates `.cdsrc-private.json` in your project root with the HANA credentials. **Never commit this file** — add it to `.gitignore`.

### Step 3 — Run locally against HANA
```bash
cds watch --profile hybrid
```

Your service runs on `http://localhost:4004` but all data and procedure calls go against the real HANA Cloud database.

> **Note:** On trial accounts, HANA Cloud shuts down overnight. If you get a connection error, start your instance from the BTP cockpit first.

### Step 4 — Call the function

In the browser or via curl:
```
GET http://localhost:4004/odata/v4/catalog/getOrderSummary(status='open')
```

```bash
curl "http://localhost:4004/odata/v4/catalog/getOrderSummary(status='open')"
```

Expected response:
```json
{
  "@odata.context": "...",
  "value": [
    { "customer_id": "C001", "customer_name": "Alice", "order_count": 2, "total_amount": 200.00 },
    { "customer_id": "C003", "customer_name": "Carol", "order_count": 1, "total_amount": 120.00 }
  ]
}
```

### Error #5: Function returns empty array `[]` despite data existing

**Symptom:**
```json
{ "value": [] }
```

**Cause:** HANA returns `OUT` parameter keys in **UPPERCASE**. If your handler uses `result.ot_results` (lowercase), it gets `undefined` which CAP serializes as an empty array.

**Diagnosis — add a log to see the raw result:**
```js
this.on('getOrderSummary', async (req) => {
  const result = await db.run(`CALL "get_order_summary"(iv_status => ?, ot_results => ?)`, [req.data.status])
  console.log('RAW RESULT:', JSON.stringify(result))  // check the actual key name
  return result.OT_RESULTS
})
```

**Fix:** Use the uppercase key name matching the OUT parameter name in your procedure:
```js
return result.OT_RESULTS  // correct — HANA uppercases all OUT parameter keys
return result.ot_results  // wrong — returns undefined → empty array
```

## 11. Key Concepts

### Automation summary

| Artifact | Who creates it |
|---|---|
| `.hdbtable` | CAP (`cds build`) |
| `.hdbview` | CAP (`cds build`) |
| `.hdbtabledata` + CSV | CAP (`cds build`) |
| `gen/db/src/gen/.hdiconfig` | CAP (`cds build`) |
| `db/src/.hdiconfig` | **You — manual, one-time** |
| `.hdbprocedure` | **You — manual** |
| `.hdbsequence`, `.hdbrole` etc. | **You — manual** |

### `db/src/` vs `db/src/gen/`

- `db/src/gen/` — owned by CAP. Regenerated on every `cds build`. Do not edit.
- `db/src/` — owned by you. Copied as-is into `gen/db/src/`. Persistent across builds.

### `.hdbtabletype` — when you need it

You only need a separate `.hdbtabletype` file when:
- Multiple procedures share the same output shape
- You call a procedure from another procedure and need a typed variable

For a single procedure with a simple output, define the table structure inline in the `OUT TABLE(...)` parameter — no separate file needed.

### Parameter binding in `CALL`

```js
// Named parameter style (recommended)
await db.run(`CALL "proc_name"(in_param => ?, out_param => ?)`, [inputValue])

// HANA returns OUT parameter keys in UPPERCASE regardless of how you named them
// result.OUT_PARAM — correct
// result.out_param — undefined
```
